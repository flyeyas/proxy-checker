# Flask 轻量重构方案

## 目标

本次重构的目标不是更换复杂技术栈，而是在保持现有功能和部署方式的前提下，把后端职责拆清楚，让后续继续维护自动任务、仓库、检测引擎和统一代理转发端口时更稳。

核心目标：

- 保持现有 API 路径不变。
- 保持 `index.html`、`app.js`、`login.html` 前端入口基本不动。
- 保持 JSON / TXT 文件存储，不在第一阶段引入数据库。
- 保持自动任务和检测任务单进程运行。
- 代理网关 `7890` 独立运行，不塞进 Flask。
- Docker 部署方式尽量保持不变。

## 推荐技术栈

```text
Flask        管理后台 API / 静态页面 / 登录
Waitress     轻量 WSGI Server，单进程多线程
自定义 TCP    继续负责 7890 统一代理转发
```

建议新增依赖：

```text
Flask
waitress
```

不要使用多进程 worker。当前项目有检测 session、自动任务 runtime、调度线程等内存态数据，多进程会导致状态分裂。推荐运行模型是一个 Python 进程内启动多个线程：

```text
一个 Python 进程
├── Flask / Waitress :8888
├── 自动任务 scheduler thread
└── 代理网关 TCP server thread :7890
```

## 启动入口

目标形态：

```python
from waitress import serve

from proxy_checker.app import create_app
from proxy_checker.config import settings
from proxy_checker.gateway.server import start_proxy_gateway
from proxy_checker.services.auto_service import start_auto_scheduler

app = create_app()

if __name__ == "__main__":
    start_auto_scheduler()
    start_proxy_gateway()
    serve(
        app,
        host="0.0.0.0",
        port=settings.port,
        threads=settings.http_threads,
    )
```

## Flask App 工厂

`proxy_checker/app.py`：

```python
from flask import Flask

from proxy_checker.http.auth_routes import auth_bp
from proxy_checker.http.settings_routes import settings_bp
from proxy_checker.http.repo_routes import repo_bp
from proxy_checker.http.auto_routes import auto_bp
from proxy_checker.http.check_routes import check_bp
from proxy_checker.http.static_routes import static_bp

def create_app():
    app = Flask(__name__)

    app.register_blueprint(static_bp)
    app.register_blueprint(auth_bp)
    app.register_blueprint(settings_bp)
    app.register_blueprint(repo_bp)
    app.register_blueprint(auto_bp)
    app.register_blueprint(check_bp)

    return app
```

## 认证层

认证逻辑从 `server.py` 拆到：

```text
services/auth_service.py
```

建议提供：

```python
def is_auth_enabled() -> bool
def make_auth_token() -> str
def verify_auth_token(token: str) -> bool
def make_auth_cookie(token: str, max_age: int) -> str
def get_request_token(request) -> str | None
def require_auth()
```

Flask 里使用装饰器：

```python
@repo_bp.post("/api/repo/save")
@require_auth
def save_repo():
    ...
```

必须保留现有行为：

- 未登录访问 `/` 返回 `login.html`。
- 未登录访问 `/app.js` 返回 `401`。
- `/api/repo/<token>.txt` 和 `/api/repo/<token>.json` 保持公开可访问。

## 服务层

HTTP 路由只负责收参、鉴权、返回 JSON，不直接写业务细节。

推荐调用关系：

```text
http routes
    -> services
        -> storage
        -> checking / gateway
```

示例：

```text
auto_routes.py
    -> auto_service.py
        -> auto_store.py
        -> repo_service.py
        -> check_service.py
```

仓库入库策略放在：

```text
services/repo_service.py
```

包括：

```python
def result_matches_policy(result, policy)
def merge_repo_results(token, repo, results, checked_inputs, policy)
def result_to_repo_item(result, existing=None)
```

代理网关读取 A/B 代理时，也应复用 repo service，避免重复筛选逻辑。

## 代理网关

代理网关不要交给 Flask 处理。Flask 适合 HTTP API，不适合底层 TCP 隧道。

拆分为：

```text
gateway/server.py
├── start_proxy_gateway()
└── ProxyGatewayHandler

gateway/upstream_pool.py
├── load_candidates()
├── normalize_upstream_proxy()
└── choose_upstream()

gateway/tunnel.py
├── relay()
├── handle_http_proxy_request()
└── handle_connect_tunnel()
```

默认配置：

```text
PROXY_GATEWAY_ENABLED=true
PROXY_GATEWAY_BIND=127.0.0.1
PROXY_GATEWAY_PORT=7890
PROXY_GATEWAY_GRADES=A,B
PROXY_GATEWAY_TOKEN=
PROXY_GATEWAY_TIMEOUT=20
```

Docker 内部可以绑定 `0.0.0.0`，但宿主机默认只映射 `127.0.0.1:7890`，避免 VPS 变成公网开放代理。

## 配置层

配置从 `server.py` 顶部拆到：

```text

```

建议使用简单 dataclass，不引入 Pydantic：

```python
from dataclasses import dataclass

@dataclass
class Settings:
    port: int
    http_threads: int
    auth_password: str
    auth_session_days: int
    check_rounds: int
    max_concurrent: int
    proxy_gateway_enabled: bool
    proxy_gateway_bind: str

```

配置来源顺序保持现状：

```text

```

## 迁移步骤

建议分 6 步执行，每一步都保证项目可运行。

### 第 1 步：建立包结构和配置层

- 新建 `proxy_checker/`。
- 新建 `config.py`。
- 把环境变量、`config.json`、`config.local.json` 的读取逻辑迁出 `server.py`。
- `server.py` 仍然可以启动旧 Handler。

验收：

- `python server.py` 能启动。
- Docker 能启动。
- `/api/capabilities` 正常。

### 第 2 步：拆 storage 层

- 迁移原子写文件逻辑。
- 迁移 repo / checked / auto / run log 的读写。
- 原 API 行为不变。

验收：

- 仓库 TXT / JSON 链接正常。
- 自动任务状态能保存。
- 日志能读写。

### 第 3 步：拆 service 层

- 迁移认证、仓库、自动任务、检测任务逻辑。
- `server.py` 的 Handler 只调用 service。

验收：

- 手动检测能运行。
- 自动任务能保存、立即运行、停止。
- 入库策略行为不变。

### 第 4 步：接入 Flask

- 新建 `create_app()`。
- 用 Blueprint 替换 `Handler.do_GET` / `Handler.do_POST`。
- 保持所有 API 路径不变。

验收：

- 前端无需修改即可正常使用。
- 登录行为一致。
- `app.js` 未登录仍返回 401。
- 仓库分享链接仍公开。

### 第 5 步：拆代理网关

- 把 `ProxyGatewayHandler` 拆到 `gateway/server.py`。
- 把上游选择逻辑拆到 `gateway/upstream_pool.py`。
- 把转发和隧道逻辑拆到 `gateway/tunnel.py`。

验收：

- `curl -x http://127.0.0.1:7890 https://example.com` 可用。
- HTTP 转发测试通过。
- CONNECT 隧道测试通过。

### 第 6 步：改用 Waitress 启动

- `server.py` 使用 `waitress.serve()` 启动 Flask。
- Docker 保持 `CMD ["python", "server.py"]`。
- 明确单进程多线程。

验收：

- Docker 健康检查通过。
- GitHub Actions 镜像构建通过。
- VPS Compose 部署方式不变。

## Docker 调整

`Dockerfile` 可以继续保留：

```dockerfile
CMD ["python", "server.py"]
```

新增环境变量：

```text
HTTP_THREADS=16
```

不要使用多 worker 配置。

## 风险点

- 自动任务不能多进程运行，否则调度器会重复跑。
- 检测任务 session 目前在内存里，多进程会丢状态。
- 代理网关不能被 Flask 接管。
- 静态资源鉴权行为必须保持一致。
- `/api/repo/<token>.txt` 和 `.json` 必须继续公开可访问。
- Docker 默认不能把 `7890` 暴露成公网开放代理。

## 最终效果

重构完成后，项目职责会变成：

```text
Flask          管 API / 静态页面 / 登录
service 层      管业务
storage 层      管数据读写
gateway 层      管统一代理转发
checking 层     管代理检测
sources 层      管免费代理源
frontend       暂时保持原生 HTML / JS
```

这套方案足够轻量，也不会把项目从“可部署的小工具”改成复杂工程。它的主要价值是把当前越来越大的 `server.py` 拆开，让后续继续加功能时更安全。
